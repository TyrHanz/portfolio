About Me

구조를 고민하는 개발자.

기능 완성 너머의 유지보수성을 고민합니다.

쟁여 프로젝트에서 마이페이지, 재고화면, 상품상세화면, 검색화면, 구매예정화면을 담당했습니다. 사용자가 반복적으로 쓰는 핵심 흐름을 구현하며 데이터 변경이 화면에 안정적으로 반영되는 구조를 고민했습니다.

처음에는 기능을 동작시키는 것 자체에 집중했지만, 화면과 기능이 늘어날수록 작은 변경이 다른 화면에 영향을 줄 수 있다는 점을 경험했습니다. 그때부터 어떻게 만들 것인가만큼 어떻게 유지할 것인가를 함께 생각하게 됐습니다.

CoreData와 RxSwift를 연결해 재고 목록 갱신 흐름을 정리하고, Coordinator와 DI로 화면 전환과 의존성 생성을 분리했습니다. 공통 UI 컴포넌트를 만들면서 재사용성뿐 아니라 사용하는 쪽의 단순함도 중요하다는 점을 배웠습니다.

  • Local First로컬 데이터를 기준으로 빠른 재고 조회와 수정 흐름을 만듭니다.
  • Observable Flow사용자 입력과 CoreData 변경을 같은 방식으로 다룹니다.
  • Readable Structure계층 분리의 목적과 실제 효용을 함께 확인합니다.

UX
UX

RxSwift 기반 검색·필터 흐름

검색어, 카테고리, 재고 상태 필터가 자주 변경되어 ViewController에서 직접 처리하면 상태값과 조건문이 늘어나는 문제가 있었습니다.

  • 해결ViewModel을 Input/Output 구조로 두고, debounce와 distinctUntilChanged로 불필요한 검색 연산을 줄였습니다.
  • 결과사용자 입력, 필터 선택, 목록 갱신을 같은 흐름으로 다루며 ViewController 역할을 UI 바인딩으로 좁혔습니다.
UIKit RxSwift ViewModel

Data

NSFRC와 RxSwift 재고 자동 갱신

재고 수량 변경, 상품 삭제, 상세 화면 상태 변경 이후 목록을 수동 reload에 의존하면 갱신 기준이 흐려질 수 있었습니다.

  • 해결CoreData 변경 감지는 NSFetchedResultsController가 맡고, 변경 이벤트를 Observable로 감싸 ViewModel에서 구독했습니다.
  • 결과저장소 변경과 화면 갱신 흐름을 분리해 재고 목록이 데이터 변경을 기준으로 안정적으로 갱신되게 했습니다.
CoreData NSFRC RxSwift
UI
Refactor

DTO와 Domain 모델 분리

CoreData Entity나 서버 Payload 구조가 ViewModel에 직접 섞이면 저장 방식이나 응답 형태 변경이 화면 로직까지 번질 수 있었습니다.

  • 해결CoreData Entity, 서버 전송용 Payload, 앱 내부 Domain 모델을 구분하고 변환 메서드로 경계를 만들었습니다.
  • 배운 점타입을 나누는 것만으로 충분하지 않고, Domain이 실제 정책과 규칙을 담아야 분리의 의미가 커진다는 점을 배웠습니다.

Architecture

Coordinator와 DI 책임 분리

재고 화면에서 상품 상세, 검색, 구매예정 화면으로 이어지는 흐름이 많아 ViewController에 화면 생성과 이동 코드가 쌓일 수 있었습니다.

  • 해결화면 이동은 Coordinator가 담당하고, ViewModel과 Repository는 외부에서 주입받도록 구성했습니다.
  • 결과ViewController는 사용자 이벤트 전달과 UI 갱신에 집중하고, 객체 간 의존성을 더 명확히 드러낼 수 있었습니다.
Coordinator DI MVVM

01 App Architecture

  • UIKit 기반 MVVM으로 화면 표현과 상태 가공 책임 분리

  • ViewModel과 Repository를 외부에서 주입해 의존성 생성 위치 정리

02 Async & State

  • ViewModel Input/Output 구조로 사용자 이벤트와 화면 상태 연결
  • 검색어 변경에 debounce, distinctUntilChanged를 적용해 불필요한 연산 감소
  • NSFetchedResultsController 변경 이벤트를 RxSwift Observable 흐름으로 연결

03 Data Modeling

  • CoreData를 기준 데이터로 둔 Local First 재고 관리 구조 선택
